### Target features

[RFC 0020] introduces a new `@[TargetFeature]` annotation that allows to specify
CPU features or a CPU model or variant for a specific function only. It
complements the `--mattr` and `--mcpu` CLI arguments that target the whole
program. This allows to embed multiple optimized functions for different CPU
features into a single executable, think SIMD for example with a portable
implementation plus AVX2 and AVX512 alternatives.

The program is responsible to detect which feature is available and to call the
proper function at runtime, otherwise the program might crash with SIGILL for
example.

For example:

```crystal
{% raw %}
{% if flag?(:x86_64) %}
  @[TargetFeature("+avx2")]
  private def foo_avx2
  end

  private def cpu_supports_avx2?
  end
{% end %}

private def foo_portable
end

def foo
  {% if flag?(:x86_64) %}
    return foo_avx2 if cpu_supports_avx2?
  {% end %}
  foo_portable
end
{% endraw %}
```

While not strictly required, we still recommend to wrap architecture specific
method definitions within macro flag checks.

- Add `@[TargetFeature]` annotation ([#16717])

_Thanks [@stakach]

[RFC 0020]: https://github.com/crystal-lang/rfcs/pull/20

### Execution contexts

Execution contexts from [RFC 0002] continue as a final final preview feature
with opt-in with compiler flags `-Dpreview_mt -Dexecution_context` and we plan
to finally enable the feature by default in the 1.21 release!

In addition to fixing bugs, we implemented M:N scheduling that allows threads to
be attached and detached from a context. It allowed us to stop blocking other
fibers on certain syscalls, such as `getaddrinfo` for example.

- Add `Fiber::ExecutionContext::ThreadPool` ([#15885], [#16750])
- Detach execution context scheduler from running thread during blocking syscall ([#15871], [#16679])

The parallel contexts now automatically start threads to adapt to the actual
workload and distribute fibers across more cores when really needed.

- Add adaptive scaling to `ExecutionContext::Parallel` ([#16719])

_This effort is part of the [ongoing project to improve multi-threading support]
with the help of [84codes]._

### Process API

This release brings a new ergonomic, safe, and portable API for spawning sub-processes, as proposed in [RFC 0025].

As a most significant change, modern API treats the command line as an array of strings.
The first element is the program to execute, and the remaining elements are its arguments.

```cr
# legacy API:
Process.run("crystal", ["tool", "format"])

# modern API:
Process.run(["crystal", "tool", "format"])
```

A convenient new addition are methods for capturing the process output (`Process.capture` and `Process.capture_result`).

The `shell` parameter is dropped without replacement. If you need shell behaviour,
we recommend running a shell explicitly.

These are the most significant individual changes:

- Add overloads with combined `args` parameter instead of `command, args` ([#16681], [#16739])
- Add `Process.run?` ([#16738])
- Add `Process.capture` ([#16773])

All new methods and overloads are experimental.
We expect to stabilize the API for the next release.

### IO uring

An io_uring event loop is available for Linux targets and can be selected by
specifying `-Devloop=io_uring` when compiling. The event loop is highly
experimental and it's execution benefits will differ for every program. We don't
expect it to be faster than epoll, maybe even slower, except maybe with on some
benchmarks with SQPOLL enabled.

SQPOLL can be selected by specifying the idle time for the kernel threads with
`-Dio_uring_sq_thread_idle=200` (for 200ms). It is recommended to keep the idle
time low (below 2s), otherwise your CPU cores will regularly spin at 100% usage
when doing nothing.

An interesting but possibly weird behavior, is that an any I/O call that _could_
yield the calling fiber will now _always_ yield. One benefit is that an always
ready socket won't block other fibers from progressing, an issue is that there
might be lots of context switches.

- Add io_uring event loop (linux) ([#16264])

_Thanks [@ysbaddaden]_

### Sendfile

An experimental API is available to support `sendfile` (UNIX) and `TransmitFile`
(Windows). The behavior is emulated on targets that don't support it (NetBSD,
OpenBSD).

- Add `Socket#sendfile` (and `Crystal::EventLoop::Socket#sendfile`) ([#16665])

_Thanks [@ysbaddaden]_

### KernelTLS

Kernel TLS is now supported by default on Linux and FreeBSD as long as support
has been compiled in OpenSSL (should be by default) and enabled on the system
(likely disabled by default), for example `modprobe tls` on Linux.

- Add Kernel TLS support to custom `OpenSSL::BIO` ([#16646])

_Thanks [@ysbaddaden]_

### Deprecations

A single deprecation this time and it's a trivial one: rename `Mutex` as
`Sync::Mutex` to use the new algorithm introduced in the Crystal 1.19 release.

We're experimenting with a two-step mechanism to deprecate a feature. First, we
soft deprecate the method in the documentation with a `DEPRECATE:` callout, then
after a few releases we'll hard deprecate by adding the `@[Deprecated]`
annotation.
